<p>This guide is intended for developers who wish to modify the firmware of the ODrive.
As such it assumes that you know things like how to use Git, what a compiler is, etc. If that sounds scary, turn around now.</p>
<p>The official releases are maintained on the <code>master</code> branch. However since you are a developer, you are encouraged to use the <code>devel</code> branch, as it contains the latest features.</p>
<p>The project is under active development, so make sure to check the <a href="/odriverobotics/ODrive/blob/devel/CHANGELOG.md">Changelog</a> to keep track of updates.</p>

<h2><a id="user-content-prerequisites" class="anchor" aria-hidden="true" href="#prerequisites"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Prerequisites</h2>
<p>The recommended tools for ODrive development are:</p>
<ul>
<li><strong>make</strong>: Used to invoke tup</li>
<li><strong>Tup</strong>: The build system used to invoke the compile commands</li>
<li><strong>ARM GNU Compiler</strong>: For cross-compiling code</li>
<li><strong>ARM GDB</strong>: For debugging the code and stepping through on the device</li>
<li><strong>OpenOCD</strong>: For flashing the ODrive with the STLink/v2 programmer</li>
<li><strong>Python 3</strong>, along with the packages <code>PyYAML</code>, <code>Jinja2</code> and <code>jsonschema</code>: For running the Python tools (<code>odrivetool</code>). Also required for compiling firmware.</li>
</ul>
<p>See below for specific installation instructions for your OS.</p>
<p>Depending on what you're gonna do, you may not need all of the components.</p>
<p>Once you have everything, you can verify the correct installation by running:</p>
<div class="highlight highlight-source-shell"><pre>$ arm-none-eabi-gcc --version
$ arm-none-eabi-gdb --version
$ openocd --version             <span class="pl-c"><span class="pl-c">#</span> should be 0.10.0 or later</span>
$ tup --version                 <span class="pl-c"><span class="pl-c">#</span> should be 0.7.5 or later</span>
$ python --version              <span class="pl-c"><span class="pl-c">#</span> should be 3.7 or later</span></pre></div>
<h4><a id="user-content-linux-ubuntu--2004" class="anchor" aria-hidden="true" href="#linux-ubuntu--2004"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Linux (Ubuntu &lt; 20.04)</h4>
<div class="highlight highlight-source-shell"><pre>sudo add-apt-repository ppa:team-gcc-arm-embedded/ppa
sudo apt-get update
sudo apt-get install gcc-arm-embedded
sudo apt-get install openocd
sudo apt-get install git-lfs

<p>To customize the compile time parameters, copy or rename the file <code>Firmware/tup.config.default</code> to <code>Firmware/tup.config</code> and edit the parameters in that file:</p>
<p><strong>CONFIG_BOARD_VERSION</strong>: The board version you're using. Can be <code>v3.1</code>, <code>v3.2</code>, <code>v3.3</code>, <code>v3.4-24V</code>, <code>v3.4-48V</code>, <code>v3.5-24V</code>, <code>v3.5-48V</code>, etc. Check for a label on the upper side of the ODrive to find out which version you have. Some ODrive versions don't specify the voltage: in that case you can read the value of the main capacitors: 120uF are 48V ODrives, 470uF are 24V ODrives.</p>
<p><strong>CONFIG_DEBUG</strong>: Defines wether debugging will be enabled when compiling the firmware; specifically the <code>-g -gdwarf-2</code> flags. Note that printf debugging will only function if your tup.config specifies the <code>USB_PROTOCOL</code> or <code>UART_PROTOCOL</code> as stdout and <code>DEBUG_PRINT</code> is defined. See the IDE specific documentation for more information.</p>
<p>You can also modify the compile-time defaults for all <code>.config</code> parameters. You will find them if you search for <code>AxisConfig</code>, <code>MotorConfig</code>, etc.</p>

<h3><a id="user-content-maintaining-modified-generated-code" class="anchor" aria-hidden="true" href="#maintaining-modified-generated-code"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Maintaining modified generated code</h3>
<p>When generating the code, STM32CubeMX will nuke everything except some special sections that they provide. These sections are marked like <code>USER CODE BEGIN</code>...<code>USER CODE END</code>.
We used to try to make sure all edits we made to the generated code would only go in these sections, so some code structrure may reflect that.
However over time we realized this will not be tenable, so instead we use git to rebase all changes of the generated code whenever we need to regenerate it.
We use two special branches that will help us to do this, they are <code>STM32CubeMX-start</code> and <code>STM32CubeMX-end</code>.
How to use these is shown in the following example.</p>
<p><strong>Note</strong>: Due to how this rebasing is done, all development that changes the generated code should be done directly on <code>STM32CubeMX-end</code>, and not based on <code>devel</code>, then follow step 4 below to carry them over to your feature branch. If you did some changes to the generated code based from <code>devel</code>, you need to cherry pick just those changes over to <code>STM32CubeMX-end</code>.</p>
<h3><a id="user-content-1-ensuring-a-clean-slate" class="anchor" aria-hidden="true" href="#1-ensuring-a-clean-slate"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>1. Ensuring a clean slate</h3>
<ul>
<li>We do all changes to the STM32CubeMX config and regenerate the code on top of <code>STM32CubeMX-start</code>.
<ul>
<li><code>git checkout STM32CubeMX-start</code></li>
</ul>
</li>
<li>Run stm32cubeMX and load the <code>Firmware/Board/v3/Odrive.ioc</code> project file.
<ul>
<li>If the tool asks if you wish to migrate to a new version, choose to download the old firmware package (unless you want to use the latest libraries)</li>
</ul>
</li>
<li>Without changing any settings, press <code>Project -&gt; Generate code</code>.</li>
<li>You may need to let it download some drivers and such.</li>
<li>STM32CubeMX may now have a newer version of some of the libraries, so there may be changes to the generated code even though we didn't change any settings. We need to check that everything is still working, and hence check in the changes:</li>
<li><code>git config --local core.autocrlf input</code> - This will tell git that all files should be checked in with LF endings (CubeMX generates CRLF endings).</li>
<li><code>git diff</code> - Ignore the pile of line ending warnings.</li>
<li>If you feel qualified: you can now ispect if CubeMX introduced something stupid. If there were any changes, and they look acceptable, we should commit them:
<ul>
<li><code>git commit -am "Run STM32CubeMX v1.21"</code> - Replace with actual version of CubeMX</li>
</ul>
</li>
</ul>
<h3><a id="user-content-2-making-changes-to-the-stm32cubemx-config" class="anchor" aria-hidden="true" href="#2-making-changes-to-the-stm32cubemx-config"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>2. Making changes to the STM32CubeMX config</h3>
<ul>
<li>After completing the above steps, make sure the working directory is clean:
<ul>
<li><code>git status</code> should include "nothing to commit, working tree clean"</li>
</ul>
</li>
<li>Make your changes in STM32CubeMX, save the project and generate the code. (<code>Project -&gt; Generate code</code>)</li>

<li>If everything looks ok, you can commit your changes.</li>
</ul>

<p>If you need to modify libfibre add <code>CONFIG_BUILD_LIBFIBRE=true</code> to your tup.config and rerun <code>make</code>. After this you can start <code>odrivetool</code> (on your local PC) and it will use the updated libfibre.</p>
<p>To cross-compile libfibre for the Raspberry Pi, run <code>make libfibre-linux-armhf</code> or <code>make libfibre-all</code>. This will require a docker container. See <a href="/odriverobotics/ODrive/blob/devel/Firmware/fibre-cpp/README.md">fibre-cpp readme</a> for details.</p>
<p>If you're satisfied with the changes don't forget to generate binaries for all
supported systems using <code>make libfibre-all</code>.</p>

<p>We use GitHub Releases to provide firmware releases.</p>
<ol>
<li>Cut off the changelog to reflect the new release</li>
<li>Merge the release candidate into master.</li>
<li>Push a (lightweight) tag to the master branch. Follow the existing naming convention.</li>
<li>If you changed something in libfibre, regenerate the binaries using <code>make libfibre-all</code>. See <a href="#modifying-libfibre">Modifying libfibre</a> for details.</li>
<li>Push the python tools to PyPI (see setup.py for details).</li>
<li>Edit the release on GitHub to add a title and description (copy&amp;paste from changelog).</li>
</ol>
<h2><a id="user-content-other-code-maintenance-notes" class="anchor" aria-hidden="true" href="#other-code-maintenance-notes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Other code maintenance notes</h2>
<p>The cortex M4F processor has hardware single precision float unit. However double precision operations are not accelerated, and hence should be avoided. The following regex is helpful for cleaning out double constants:
find: <code>([-+]?[0-9]+\.[0-9]+(?:[eE][-+]?[0-9]+)?)([^f0-9e])</code>
replace: <code>\1f\2</code></p>

<h2><a id="user-content-notes-for-contributors" class="anchor" aria-hidden="true" href="#notes-for-contributors"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Notes for Contributors</h2>
<p>In general the project uses the <a href="https://google.github.io/styleguide/cppguide.html" rel="nofollow">Google C++ Style Guide</a>, with a few exceptions:</p>
<ul>
<li>The default indentation is 4 spaces.</li>
<li>The 80 character limit is not very strictly enforced, merely encouraged.</li>
<li>The file extensions *.cpp and *.hpp are used instead of *.cc and *.h.</li>
</ul>
<p>Your help is welcome! However before you start working on a feature/change that will take you a non-negligible amount of time and that you plan to upstream please discuss your plans with us on GitHub or Discord. This will ensure that your implementation is in line with the direction that ODrive is going.</p>
<p>When filing a PR please go through this checklist:</p>
<ul>
<li>Make sure you adhere to the same coding style that we use (see note above).</li>
<li>Update CHANGELOG.md.</li>
<li>If you removed/moved/renamed things in <code>odrive-interface.yaml</code> make sure to add corresponding bullet points tp the "API migration notes" section in the changelog. Use git to compare against the <code>devel</code> branch.</li>
<li>Also, for each removed/moved/renamed API item use your IDE's search feature to search for occurrences of this name. Update the places you found (this will usually be documentation and test scripts).</li>
<li>If you added things to <code>odrive-interface.yaml</code> make sure the new things have decent documentation in the YAML file. We don't expect 100% coverage but use good sense of what to document.</li>
<li>Make sure your PR doesn't contain spurious changes that unnecessarily add or remove whitespace. These add noise and make the reviewer's lifes harder.</li>
<li>If you changed any enums in <code>odrive-interface.yaml</code>, make sure you update <a href="/odriverobotics/ODrive/blob/devel/tools/odrive/enums.py">enums.py</a>. The file includes instructions on how to do this. Check the diff to verify that none of the existing enumerators changed their value.</li>
</ul>
